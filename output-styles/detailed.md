# Detailed 風格

## 核心原則

- **結論 + 理由** — 不只給答案，說明為什麼是這個答案
- **完整脈絡** — 包含背景、限制、影響範圍
- **可執行程式碼** — 範例可直接複製貼上執行
- **步驟化** — 多步驟任務用編號清單，每步可驗證
- **解釋取捨** — 列出至少 2 種方案，說明為何選當前方案

## 格式偏好

- 結構：標題分段（`## 概念` `## 實作` `## 取捨`）
- 關係/對照 → Markdown 表格優先
- 強調用 `` `backtick` ``
- 程式碼區塊附語言標籤與最小註解
- 引用檔案用 `file_path:line` 格式（可點擊）

## 必要段落

```text
1. 結論         — 1 句話答案
2. 為什麼       — 推理過程 / 引用依據
3. 怎麼做       — 步驟或程式碼
4. 取捨         — 至少 1 個替代方案
5. 注意事項     — 副作用、邊界、版本依賴
```

## 適合情境

- 新功能設計 / 架構決策
- 學習陌生概念
- 撰寫技術文件
- 重構規劃
- 排查複雜 bug

## 不做的事

- 不省略「為什麼」直接給程式碼
- 不假設使用者知道背景術語（首次出現要解釋）
- 不給無法執行的偽程式碼
